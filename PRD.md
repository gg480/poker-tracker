# 德扑积分榜 (Poker Tracker) 产品需求文档

**版本**: 3.0
**日期**: 2026-04-26
**产品形态**: 跨平台App（Tauri / React Native）
**状态**: 正式版

---

## 1. 执行摘要

德扑积分榜是一款面向朋友局德州扑克玩家的**多人协作式**积分追踪与分析工具。通过数字化记录比赛积分、支持各人自行录入、实时汇总、智能清分提醒，帮助玩家摆脱繁琐的手动记账，专注于游戏本身的乐趣。

**核心价值主张**：

> **朋友局积分神器 — 人人可录入、局局能汇总、分享超便捷**

**三大核心差异点**：
1. **多人协作录入**：各人自行录入自己的积分，系统实时汇总，减少记录员负担
2. **一键分享战绩**：当局结束后生成分享图片，发到群里大家都能看
3. **数据主权**：本地存储，无需注册账号，数据完全自主

## 2. 问题陈述

### 2.1 用户场景还原

```
⏰ 比赛前
- 玩家陆续到场，有人问："今天谁积分最高来着？"
- 有人掏出手机打开App查看历史积分

✏️ 比赛中（核心场景）
- 每场结束后：
  - 以前：用微信在线表格填写，经常填错、覆盖、找不到

  - 一键生成本局积分截图，发到群里

- "我积分超过8000了吧？该请吃饭了"
- App自动检测并提醒当事人
- 记录员确认清分金额，系统自动结算

📊 赛季结束
- 积分封存，新赛季从0开始
- 生成赛季总结报告，分享到群里

📚 复盘分析
- 查看全部赛季历史数据
- 穿透分析特定赛季
- 记录经典手牌，对照某天对局
- 手牌GTO分析点评
```

## 5. P0 核心功能详情

### 5.1 多人协作录入

**功能描述**：
每场结束后，各玩家自行在App中录入自己的积分，系统实时汇总并校验。

**用户故事**：
```
作为 记录员
我想要 每个人自己录入自己的积分
以便于 减少单点录入的错误和负担

验收标准：
- 进入"录入"页面，选择当前场次
- 每个玩家输入自己的积分（正数赢，负数输）
- 系统实时显示所有玩家的录入状态（已录入/未录入）
- 录入完成后点击"汇总"，系统校验合计是否为零
- 合计不为零时提示差额，不允许提交
- 汇总成功后生成当局战绩
```

**交互流程**：
```
┌─────────────────────────────────────┐
│  📅 2026-04-26 第5场                │
├─────────────────────────────────────┤
│  状态：3/5人已录入                 │
│                                     │
│  ✅ 佳  +500                        │
│  ✅ 茄  -300                        │
│  ⏳ 润  待录入                      │
│  ⏳ 谦  待录入                      │
│  ⏳ 卢老师  待录入                   │
│                                     │
│  合计：+200  ⚠️ 不为零，差额200    │
│                                     │
│       [ 等待剩余玩家录入  ]          │
└─────────────────────────────────────┘

所有玩家录入后：
┌─────────────────────────────────────┐
│  📊 第5场汇总                       │
│                                     │
│  佳  +500  茄  -300                │
│  润  +200  谦  -400                │
│                                     │
│  卢老师  0（平和）                  │
│                                     │
│  合计：0  ✅                        │
│                                     │
│  [📤 生成分享图]  [💾 确认保存]     │
└─────────────────────────────────────┘
```

**技术要求**：
- 支持离线录入，本地暂存
- 数据通过跨平台同步机制汇总（局域网广播 / 云端同步可选）
- 汇总结果需要所有当事人确认

### 5.2 实时汇总校验

**功能描述**：
系统自动校验所有玩家积分合计是否为零，确保数据准确性。

**规则**：
| 场景 | 处理方式 |
|------|----------|
| 合计为零 | 允许提交，显示✅ |
| 合计不为零 | 显示差额，禁止提交 |
| 有人未录入 | 显示等待状态 |
| 超时未录入 | 提醒当事人，或记录员代填 |

### 5.4 智能清分雷达

**功能描述**：
当玩家余额达到清分阈值（8000分）时，系统自动检测并提醒。

**清分阈值规则**：
| 阶段 | 触发条件 | 通知内容 |
|------|----------|----------|
| 预警 | 余额≥6400 (80%) | "⚠️ 佳的余额已达6500分，距离清分还差1500分" |
| 触发 | 余额≥8000 (100%) | "🚨 佳的余额已达8500分，请尽快清分！" |
| 催办 | 每48小时未清分 | "🔔 佳的清分还未完成，记得请吃饭哦~" |
| 完成 | 清分确认后 | "✅ 佳已清分1000分，当前余额7500分" |

**余额计算公式**：
```
balance = total_score - settle_score + season_adjust
```

### 5.5 赛季积分全清机制

**功能描述**：
赛季结束时，所有玩家积分封存，新赛季从0开始。

**流程**：
```
赛季结束 → 系统自动结算 → 积分封存 → 提示备份 → 新赛季开启
```

**封存内容**：
- 赛季总积分
- 赛季清分记录
- 赛季获奖记录
- 赛季场次统计

**新赛季**：
- 所有玩家积分重置为0
- 清分记录延续（跨赛季累计请吃饭）
- 奖项统计重置

## 6. P1 功能详情

### 6.1 腾讯文档集成

**功能描述**：
支持读取腾讯文档在线表数据，自动同步到本地App，实现数据的双向流通。

**用户故事**：
```
作为 赛季管理员
我想要 从腾讯文档在线表同步历史数据
以便于 快速导入历史记录，避免手动录入

验收标准：
- 支持输入腾讯文档在线表链接
- 自动解析表格结构，识别日期、玩家、积分
- 支持选择性导入特定日期范围的数据
- 导入后自动校验合计是否为零
- 支持将本地数据导出到腾讯文档
```

**交互流程**：
```
┌─────────────────────────────────┐
│  📋 腾讯文档同步                │
├─────────────────────────────────┤
│  🔗 文档链接: [输入框]          │
│  📅 日期范围: [选择器]          │
│  ⚙️ 同步选项:                   │
│  [ ] 导入到当前赛季             │
│  [ ] 新建赛季导入               │
│  [ ] 覆盖本地数据               │
│  [ ] 导出到文档                 │
├─────────────────────────────────┤
│  [🔄 开始同步]                 │
└─────────────────────────────────┘
```

**技术要求**：
- 支持腾讯文档API或表格导出功能
- 支持表格结构识别和数据解析
- 网络请求失败时支持重试机制
- 数据同步过程中显示进度条
- 同步结果生成详细报告

**表格结构要求**：
| 列名 | 说明 | 示例 |
|------|------|------|
| 日期 | 比赛日期 | 2026-04-26 |
| 场次 | 场次编号 | 1 |
| 玩家 | 玩家名称 | 佳 |
| 积分 | 积分（正负） | +500 |
| 赛季 | 赛季名称 | 赛季1 |

## 7. P2 功能详情：手牌记录与GTO分析

### 7.1 手牌记录

**功能描述**：
手动记录经典手牌，包括手牌信息、行动路线、结果，可对照到某天的对局中。

**用户故事**：
```
作为 玩家
我想要 记录一局经典的手牌
以便于 赛后复盘、学习提高

验收标准：
- 可以选择关联到某天某场对局
- 可以手动输入或拍照上传手牌截图
- 记录下每个玩家的行动路线
- 记录手牌结果和积分变化
```

**手牌记录内容**：
| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | UUID主键 |
| date | string | 记录日期 |
| gameId | string? | 关联的场次ID（可选） |
| players | string[] | 参与玩家 |
| handType | string | 手牌类型（AA/KK/QQ/ATs...） |
| board | string | 公共牌（flop/turn/river） |
| actions | JSON | 行动路线 [{player, action, amount}] |
| result | number | 积分变化 |
| winner | string | 赢家 |
| notes | string? | 备注 |
| tags | string[] | 标签（ bluff/ value/ hero call...） |
| photo | string? | 手牌截图（本地路径） |
| createdAt | timestamp | 创建时间 |

**行动路线记录**：
```json
{
  "player": "佳",
  "street": "flop",
  "action": "bet",
  "amount": 200,
  "hand": "AA"
},
{
  "player": "茄",
  "street": "flop",
  "action": "fold",
  "hand": "KK"
}
```

### 7.2 手牌GTO分析

**功能描述**：
将手牌数据发送给GTO Solver进行分析，获取最优策略建议和点评。

**分析内容**：
| 模块 | 内容 |
|------|------|
| 基础分析 | 手牌强度、最优行动建议 |
| GTO对比 | 与GTO推荐策略的对比 |
| EV分析 | 期望值计算 |
| 改进建议 | 指出可以改进的地方 |

**技术方案**：
- 本地GTO引擎（需集成专业德扑GTO库）
- 或调用第三方GTO API（可选，需要网络）

**注意**：
GTO分析为高复杂度功能，v2.0规划，需要研究具体技术实现方案。

|------|------|------|
| id | string | UUID主键 |
| player | string | 玩家名称 |
| seasonId | string | 所属赛季 |
| settleScore | number | 请吃饭累计积分（≥0） |
| seasonAdjust | number | 赛季调整（可正可负） |
| updatedAt | timestamp | 更新时间 |

#### HandRecord（手牌记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | UUID主键 |
| date | string | 日期 (YYYY-MM-DD) |
| seasonId | string | 所属赛季 |
| sessionId | string? | 关联场次（可选） |
| players | string | 参与玩家（逗号分隔） |
| handType | string | 手牌类型 |
| board | string | 公共牌 |
| actions | string | 行动路线（JSON字符串） |
| result | number | 积分变化 |
| winner | string | 赢家 |
| notes | string? | 备注 |
| tags | string | 标签（逗号分隔） |
| photo | string? | 手牌图片路径 |
| gtoAnalysis | string? | GTO分析结果 |
| createdAt | timestamp | 创建时间 |

### 8.2 计算公式

```
余额 = 赛季累计积分 - 请吃饭累计 + 赛季调整

赛季结束自动执行：
赛季调整 += -余额  // 余额归零
赛季封存标志 = true
新赛季开启：所有玩家积分重置为0（请吃饭累计延续）
```
